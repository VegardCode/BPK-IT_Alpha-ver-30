# Конструктор аналитических выборок · Минфин Амурской области

MVP-прототип к кейсу **«Интеллектуальный отбор данных. Конструктор аналитических выборок для бюджетного планирования»** (БФТ, Минфин АО).

Объединяет данные **трёх ключевых информационных систем (АЦК-Планирование / АЦК-Финансы / АЦК-Госзаказ)** и выгрузки **БУАУ** в едином интерфейсе. Связка идёт через сквозную бюджетную классификацию (КФСР · КЦСР · КВР · КОСГУ · КВФО).

## Возможности MVP

| Функция | Реализовано |

| Сквозная связка данных по бюджетной классификации | ✅ |
| Конструктор выборок: объект (бюджет, КФСР, КЦСР, КВР), показатели, период | ✅ |
| Поиск по наименованиям программ / мероприятий | ✅ |
| 9 показателей: план, БО, остаток, касса, соглашения, контракты, платежи, БУАУ-выплаты, БУАУ-исполнение | ✅ |
| Стратегия расчёта: «последний снимок» (нарастающий итог) и «сумма за период» | ✅ |
| Режимы агрегации: по объектам / по снимкам (timeseries) | ✅ |

## Стек

- **Backend**: Node.js 18+ · Express · csv-parse · ExcelJS
- **Frontend**: Vue 3 (CDN) · TailwindCSS (CDN) · Chart.js (CDN)
- **Хранение**: in-memory кеш CSV (для MVP)

> Целевая архитектура (по презентации БФТ): PostgreSQL Pro + Python/Django (или Java/Spring Boot) + Apache NiFi для ETL + РЕД ОС / Astra Linux. Текущий MVP легко мигрируется: модель данных и движок выборки изолированы в `server/queryEngine.js` и `server/parsers/*`.

## Запуск

Требуется **Node.js 18+**.

```bash
# Установка зависимостей
npm install

# Старт сервера (источник данных по умолчанию — папка кейса в корне)
npm start

# Или с переопределением пути к данным:
DATA_DIR="C:\path\to\Кейс_..." PORT=3000 npm start
```

Открыть в браузере: <http://localhost:3000>

## Сценарий демо

1. Главный экран — конструктор выборок.
2. **Объект**: выбрать бюджет (например, «Областной бюджет Амурской области»), при необходимости уточнить КФСР / КЦСР / КВР или ввести поиск по наименованию.
3. **Показатели**: кликнуть по нужным «пиллам» (по умолчанию выбраны план, БО, касса, контракты, платежи).
4. **Период**: выбрать вручную или применить пресет («Весь период» / «Последний снимок» / «2025 год» / «2026 год»).
5. **Стратегия**:
    - *Последний снимок (нарастающий)* — для РЧБ берётся последний снимок ≤ верхней границы периода. Корректно для нарастающих итогов.
    - *Сумма за период* — суммирует все наблюдения в окне.
6. Нажать **«Получить выборку»**. Появятся KPI-карточки с тоталами и таблица по объектам.
7. Переключить вид «Таблица / График».
8. Нажать **«Экспорт в Excel»** — будет загружен `.xlsx` с двумя листами (данные + параметры).
9. Вкладка **«Сравнение периодов»** — сравнить два временных среза (Δ абс. и %).

## API (для интеграций)

| Метод | Путь | Назначение |
|---|---|---|
| GET  | `/api/health`            | состояние и статистика загруженных данных |
| GET  | `/api/indicators`        | список показателей и классификаторов |
| GET  | `/api/snapshots`         | доступные «снимки» (даты) по источникам |
| GET  | `/api/objects?q=&limit=` | индекс уникальных объектов (бюджет + КФСР + КЦСР + КВР) |
| GET  | `/api/dictionary/:field` | словарь значений (`budget`, `kfsr`, `kcsr`, `kvr`, `kosgu`, `kvfo`, `kvsr`) |
| POST | `/api/query`             | выборка `{ indicators, filter, from, to, mode, strategy }` |
| POST | `/api/compare`           | сравнение `{ indicators, filter, periods[], strategy }` |
| POST | `/api/export/xlsx`       | выгрузка выборки в `.xlsx` |
| POST | `/api/export/compare-xlsx` | выгрузка сравнения в `.xlsx` |

### Пример запроса

```bash
curl -X POST http://localhost:3000/api/query \
  -H 'Content-Type: application/json' \
  -d '{
    "indicators": ["plan", "bo", "cash", "contracts", "payments"],
    "filter": { "budget": "", "kfsr": "", "kcsr": "" },
    "from": "2025-01-01",
    "to":   "2025-12-31",
    "mode": "aggregate",
    "strategy": "latest"
  }'
```

## Модель данных и сквозная кодировка

Каждая запись после парсинга нормализуется к единой модели:

```
{
  source:      'rchb' | 'buau' | 'agreements' | 'gz',
  snapshot:    'YYYY-MM-DD',     // дата снимка / контракта / платежа
  budget:      'Бюджет ...',
  kfsr, kcsr, kvr, kvsr, kosgu, kvfo,   // сквозные коды
  kfsrName, kcsrName, kvrName, kvsrName, kosguName, kvfoName,
  // метрики (только релевантные для источника):
  plan, bo, limit_remainder, cash,      // РЧБ
  agreements,                            // Соглашения
  contracts, payments,                   // ГЗ
  buau_payments, buau_executed           // БУАУ
}
```

Группировка «объекта» по умолчанию: `(budget, kfsr, kcsr, kvr)`.
Это даёт ~191 уникальный объект на текущем тестовом наборе.

## Особенности парсинга

- **РЧБ** — UTF-8 BOM, разделитель `;`, **первые 10 строк — шапка отчёта** (название органа, период, дата печати, фильтры). Парсер автоматически находит строку с заголовками колонок («Бюджет;Дата проводки;...») и пропускает шапку. Числа в формате `1 234 567,89` (пробелы как разделители тысяч, запятая как десятичный).
- **БУАУ** — UTF-8 BOM, разделитель `;`, заголовок в первой строке. Строка «Итого» исключается.
- **Соглашения** — UTF-8, разделитель `,`, кавычки. Дата снимка извлекается из имени файла (`на01012026.csv` → `2026-01-01`, `01012025-01042026.csv` → `2026-04-01`).
- **Госзаказ** — три связанные CSV (бюджетные строки, контракты, платежки) объединяются по `con_document_id`. Классификация подтягивается из «Бюджетные строки.csv».

## Что в нём для бизнес-заказчика

- **Без обучения** — UX в формате «выберите объект → отметьте показатели → задайте период → получите результат». 4 шага.
- **Любой пользователь** — нет SQL, нет программирования. Только селекты, кнопки и переключатели.
- **Любой срез** — данные доступны и по одному объекту, и по бюджету, и по программе целиком.
- **Готовая Excel-выгрузка** — для отчётов и руководству.
- **Сравнение периодов** — для оценки динамики (например, «01.01.2025 vs 01.01.2026»).
- **Графики** — быстрая визуализация структуры расходов / динамики во времени.
